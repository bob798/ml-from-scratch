# 视频分析行业术语速查（管理者视角）

> 给非技术管理者 / 售前 / 业务的术语扫盲表。
> 不追求技术深度，追求"听到能反应过来 + 看到陷阱"。
> 关联：[ai-video-analytics.md](ai-video-analytics.md)（场景方案）。

## 一、视频接入与协议

### RTSP（Real Time Streaming Protocol，实时流传输协议）

- **大白话**：摄像头吐视频流的"标准接口"，最通用。
- **业务含义**：90% 的摄像头都支持 RTSP，意味着 AI 服务器可以无门槛接入存量摄像头。
- **常见陷阱**：老旧摄像头 RTSP 不稳定（断流、延迟漂移）—— 部署前要测试稳定性。

### GB/T 28181（国标 28181）

- **大白话**：中国国标的视频联网协议，**政府/公安/智慧城市项目必备**。
- **业务含义**：客户如果是政府/国企，几乎一定要求支持 GB28181，否则平台对接不进去。
- **管理者关注点**：售前问客户"是否要对接现有视频平台"，对接的多半是 GB28181。

### ONVIF

- **大白话**：国际标准的摄像头互操作协议（西方版的 GB28181）。
- **业务含义**：海外项目或外资客户场景必备。

### NVR / DVR

- **NVR**（Network Video Recorder，网络录像机）：接收网络摄像头视频并存储。
- **DVR**（Digital Video Recorder，数字录像机）：接收模拟摄像头视频并存储。
- **业务含义**：客户机房里的"录像柜子"，AI 分析可以从 NVR 拉流（不直接连摄像头）。
- **管理者关注点**：从 NVR 拉流可以减少现网改造，但拉多了会拖垮 NVR。

### PTZ（Pan-Tilt-Zoom，云台摄像头）

- **大白话**：能左右转 + 上下俯仰 + 远近变焦的摄像头。
- **业务含义**：场景识别更难（画面在动），算法要做"防抖 + 重新建立 ROI"。
- **常见陷阱**：客户摄像头是 PTZ，但你的算法是按固定画面训练的 → 误报飙升。

## 二、算力与硬件

### "路"（视频流数量）

- **定义**：1 路 = 1 个独立视频流。
- **不等于摄像头台数**：1 台鱼眼可拆 4 路；1 路可同时跑多个算法。
- **关键公式**：实际算力消耗 = 路数 × 每路算法数 × 算法复杂度。
- **常见陷阱**：标称"16 路盒子"在客户跑复合算法时只能跑 5-6 路。
- **管理者关注点**：报价前必须精确测算"客户场景下每路的真实消耗"，预留 30% 余量。

### TOPS（Tera Operations Per Second）

- **大白话**：算力单位，"每秒能做多少万亿次运算"，类似 CPU 主频。
- **业务含义**：选 AI 服务器规格时的核心指标。1 路通用检测大约消耗 0.5-2 TOPS。
- **管理者关注点**：标称 TOPS 是峰值，**实际工程跑出来通常只有标称的 50-70%**。

### GPU / NPU / TPU

| 名词 | 全称 | 谁家用 | 特点 |
|------|------|--------|------|
| GPU | Graphic Processing Unit | NVIDIA（Jetson）、AMD | 通用、生态最全 |
| NPU | Neural Processing Unit | 华为（Atlas）、寒武纪、算能 | 专为 AI 设计、国产化方案 |
| TPU | Tensor Processing Unit | Google | 云端为主，国内少见 |

- **管理者关注点**：政府/国企项目优先 NPU（国产化要求），商业项目优先 GPU（生态成熟）。

### 主流边缘 AI 芯片

| 芯片 | 厂商 | 典型应用 |
|------|------|---------|
| Jetson Orin | NVIDIA | 主流商业项目 |
| Atlas 500 | 华为 | 国产化、智慧城市 |
| BM1684/1688 | 算能 | 国产化、性价比 |
| MLU220/270 | 寒武纪 | 国产化、电信/能源 |

## 三、模型与精度

### FP32 / FP16 / INT8

- **大白话**：模型参数的"精度位数"。FP32 最准但最慢，INT8 最快但精度略降。
- **业务含义**：边缘部署几乎都用 FP16 或 INT8，速度能翻 1-4 倍。
- **管理者关注点**：算法工程师说"我们用 FP32 部署"= 算力浪费严重 / 工程水平差。

### 量化（Quantization）

- **大白话**：把 FP32 模型压缩成 FP16 或 INT8 的过程。
- **业务含义**：量化好坏直接决定边缘部署成本（量化失败可能精度掉 10%+）。
- **管理者关注点**：必问"量化后精度损失多少？" 答不上来 = 没做过真上线。

### TensorRT / OpenVINO / MNN

- **TensorRT**：NVIDIA GPU 的推理加速框架（最主流）。
- **OpenVINO**：Intel 的推理加速框架。
- **MNN / NCNN**：阿里/腾讯的轻量级框架，移动端常用。
- **业务含义**：用了这些 = 工程严肃；只用 PyTorch 直接部署 = 算力浪费 50%+。

### Backbone / Head / Anchor

- **Backbone**：模型"主干"（如 ResNet、Swin），负责提取特征。
- **Head**：模型"任务头"，负责输出（检测框、分类等）。
- **Anchor**：检测模型中预设的"候选框"。
- **管理者关注点**：不必懂细节，但听到团队讨论这些 = 在做严肃工程。

## 四、工程概念

### ROI（Region of Interest，关注区域）

- **大白话**：在画面上画一个框，只分析框里的内容。
- **业务含义**：节省 20-40% 算力 + 大幅降低误报率。
- **管理者关注点**：每个摄像头都该手动标 ROI，没标 = 工程偷懒。

### 跳帧 / 抽帧

- **大白话**：摄像头 25fps，AI 只需要每秒分析 2-5 帧（垃圾、入侵不会 1 秒消失）。
- **业务含义**：算力消耗直接降到 1/5 - 1/10。
- **常见陷阱**：行为识别（如摔倒、打架）需要时序信息，不能简单跳帧。

### ISP（Image Signal Processor，图像信号处理器）

- **大白话**：摄像头硬件里的图像处理芯片（曝光、白平衡、降噪）。
- **业务含义**：ISP 调得好 = AI 检测精度更高（垃圾算法学的是 ISP 处理后的图）。
- **常见陷阱**：不同品牌摄像头 ISP 差异大，**A 厂训练的模型放 B 厂摄像头精度可能掉 20%**。

### Badcase（坏样本）

- **大白话**：模型预测错的案例。
- **业务含义**：badcase 分析是 AI 工程的核心日常工作，不做 = 模型永远不会进步。
- **管理者关注点**：每周要看团队的 badcase 分析量。

### 误报率 / 漏检率

| 指标 | 含义 | 业务后果 |
|------|------|---------|
| **误报率**（False Positive） | 本来没事，AI 报了 | 安保人员被无效告警淹没 |
| **漏检率**（False Negative） | 本来有事，AI 没报 | 真实事件错过，背锅 |

- **常见陷阱**：客户合同写"准确率 95%" → 模糊概念，必须细化成召回率 + 精确率，否则验收扯皮。

### SDK vs API

- **SDK**（Software Development Kit）：算法以库的形式给客户，**集成进客户系统**。
- **API**：算法以接口形式提供，**客户调用我们的服务**。
- **业务含义**：SDK 多用于私有化部署（合规客户）；API 多用于云端服务（轻量客户）。
- **管理者关注点**：SDK 维护成本高（要适配客户系统），API 维护成本低但有合规风险。

### 数据闭环（Data Flywheel）

- **大白话**：上线 → 收集 badcase → 重新标注 → 微调模型 → 再上线，循环往复。
- **业务含义**：**有数据闭环 = 真正的 AI 护城河**；没有 = 一次性买卖。
- **管理者关注点**：投资人/客户问"你们的数据闭环怎么做"——答不上来基本没机会。

## 五、业务指标

### 召回率（Recall）/ 精确率（Precision）

- **召回率**：该抓的抓到了几个（漏检的反面）。
- **精确率**：抓到的对不对（误报的反面）。
- **二者矛盾**：调高一个会降低另一个，需根据业务取舍。

**业务场景的优先级**：
- 入侵检测：召回优先（宁可错抓不放过）
- 垃圾检测：精确优先（宁可漏报不烦人）
- 烟火检测：召回优先到极致（漏报 = 灾难）

### POC / MVP / Pilot

- **POC**（Proof of Concept）：概念验证，证明"能做到"。通常 4-6 周。
- **MVP**（Minimum Viable Product）：最小可用产品，能上线给客户用。通常 3 个月。
- **Pilot**：试点，在客户的小规模环境跑通。
- **常见陷阱**：客户说要 POC 但不付费 → 大概率是想白嫖，要警惕。

### SLA（Service Level Agreement，服务等级协议）

- **大白话**：合同里写死的服务承诺（误报率、响应时间、可用性）。
- **业务含义**：SLA 是项目验收的法律依据，**写得越具体越对自己有利**。
- **常见陷阱**：合同写"AI 准确率 ≥ 95%" → "准确率"没有共同定义 = 永远验收不了。正确写法："在 XX 场景的测试集上召回率 ≥ 90% 且精确率 ≥ 85%"。

## 六、行业黑话速查表（听到这些词怎么反应）

| 团队/供应商说... | 你应该反应... |
|----------------|--------------|
| "我们准确率 99%" | 问：在什么数据集上？召回还是精确？ |
| "我们用了最新的 Transformer" | 问：开源 YOLO 跑这个场景能到多少？为啥不用？ |
| "我们能跑 64 路" | 问：跑什么算法的 64 路？我们这个场景实测多少路？ |
| "模型已经 SOTA 了" | 问：上线后误报率多少？badcase 怎么处理？ |
| "等数据齐了我们就能做" | 问：数据采集计划是什么？谁来标注？多久能齐？ |
| "客户系统老，对接很难" | 问：用 RTSP 拉流不行吗？GB28181 接口有没有？ |
| "这是行业惯例" | 问：能给我看一个用过这个方案的客户案例吗？ |
| "短期就能上线" | 问：上线 = demo 还是真正业务跑通？验收标准是什么？ |

---

## 一句话总结

> **不需要懂这些术语的底层原理，需要的是"听到能识别、看到能反应"。**
> **大部分忽悠不在概念本身，而在"用对的词包装错的事实"。**
> **多问一句"在我们这个场景下具体是什么数？"，能识破 80% 的水分。**

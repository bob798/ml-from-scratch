# AI 视频分析：业务与技术双视角拆解

> 场景：AI 视频分析服务器接入现有视频流（不换摄像头），部署入侵检测、异常行为识别、遗留物检测、暴露垃圾检测等 Agent，7×24 主动分析，秒级推送告警，误报率 ≤ 5%。
>
> 关联：本篇是场景方案拆解。要不要做、怎么判断该做不该做，见 [../decision/ai-video-analytics-go-no-go.md](../decision/ai-video-analytics-go-no-go.md)。

---

## 一、业务角度拆解

### 1. 客户与场景画像

| 维度 | 描述 |
|------|------|
| 目标客户 | 物业公司、园区/写字楼运营方、商业综合体、街道/城管、景区、医院、学校 |
| 核心买单人 | 安保负责人、物业总监（KPI：投诉率、事件响应时效）|
| 使用人 | 一线安保、监控室坐班人员、巡检员 |
| 现状痛点 | 1) 监控墙人工盯屏，1 人盯 20+ 路 → 漏检率高<br>2) 巡检靠走，10–30 分钟一轮，覆盖不全<br>3) 事件回溯靠翻录像，效率极低<br>4) 安保人力成本逐年上涨 |

### 2. 价值主张

一句话讲给客户听：

> 不换摄像头、不动现网，加一台盒子，让所有摄像头 7×24 自己看，秒级报警，误报率 ≤ 5%。

- **降本**：减少巡检 / 监控值班人力 30–50%
- **增效**：响应时效从 10–30 min → 秒级 / 分钟级
- **可追溯**：全程留痕归档，事件回溯从"翻录像"变成"查告警库"
- **低门槛**：复用存量摄像头（节省 70% 改造成本）

### 3. 核心场景优先级（按 ROI 排序）

| 场景 | 业务价值 | 落地难度 | 推荐优先级 |
|------|---------|---------|----------|
| 暴露垃圾检测 | 城管/物业刚需，KPI 直接挂钩 | 中（需场景化数据）| P0 |
| 入侵检测（周界、禁区）| 安防刚需 | 低（成熟方案）| P0 |
| 遗留物检测 | 安防刚需（防爆、防丢）| 中（误报控制难）| P1 |
| 异常行为识别（摔倒、打架、聚集）| 高价值但场景碎 | 高（精度难做）| P2 |

**建议打包销售但分阶段交付**：入侵 + 垃圾 + 遗留物作为 v1.0，异常行为作为 v2.0 升级项。

### 4. 商业模式建议

```
硬件：AI 分析服务器（一次性，按路数/算力规格定价）
  ├── 4 路入门版 / 16 路标准版 / 64 路企业版
软件：Agent 订阅（按场景、按年）
  ├── 基础包：入侵 + 遗留物
  └── 增值包：垃圾检测、行为识别
服务：实施部署 + 模型微调 + 运维
```

### 5. 关键业务指标（合同 SLA 候选）

| 指标 | 目标值 | 备注 |
|------|--------|------|
| 有效告警误报率 | ≤ 5% | 必须场景化定义"有效"|
| 端到端响应时延 | ≤ 3 秒（推流 → 推送）| 网络好的情况 |
| 检测召回率 | ≥ 90% | 关键场景如入侵 |
| 系统可用性 | ≥ 99.5% | 含离线兜底 |

## 二、技术角度拆解

### 1. 整体架构

```
[现有摄像头] --RTSP / GB28181-->
                   |
                   v
        +--------------------------+
        |   AI 视频分析服务器        |    <- 边缘部署
        |  +---------------------+ |
        |  | 视频解码模块         | |    (硬解码 NVDEC / VPU)
        |  +---------------------+ |
        |  | 抽帧 + 预处理        | |    (跳帧策略，2-5fps)
        |  +---------------------+ |
        |  | 多模型推理引擎       | |    (TensorRT / OpenVINO)
        |  |  - 入侵 Agent       | |
        |  |  - 垃圾 Agent       | |
        |  |  - 遗留物 Agent     | |
        |  |  - 行为 Agent       | |
        |  +---------------------+ |
        |  | 告警决策层           | |    (去重、抑制、置信度过滤)
        |  +---------------------+ |
        +-----------+--------------+
                    |
        +-----------+-----------+
        v                       v
  [告警推送服务]          [证据归档服务]
  APP / 企微 / 钉钉      视频片段 + 图片 + 元数据
                         (对象存储 / MinIO)
```

### 2. 关键技术选型

| 模块 | 选型 | 选择理由 |
|------|------|---------|
| 视频接入 | RTSP / GB28181 / ONVIF | 国标兼容存量设备，零改造 |
| 推理硬件 | NVIDIA Jetson Orin / 华为 Atlas 500 / 算能 BM1684 | 国产化可选；按路数选规格 |
| 检测模型 | YOLOv8 / RT-DETR（垃圾、遗留物、入侵）| 速度+精度均衡，工业成熟 |
| 行为识别 | SlowFast / VideoMAE / 关键点+规则 | 行为类需时序模型 |
| 推理框架 | TensorRT / OpenVINO / MNN | 边缘端推理加速，FP16/INT8 量化 |
| 任务调度 | 自研轻量调度器 / DeepStream | 多路视频流并发管理 |
| 告警去重 | 时空抑制（同目标 N 秒内只告警一次）| 控制误报核心手段 |
| 归档存储 | MinIO + PostgreSQL + 时序库 | 视频片段 + 结构化元数据 |

### 3. 暴露垃圾检测的技术难点

这个场景比通用目标检测**难得多**：
- **类别开放**：垃圾形态千变万化（袋装、散落、液体、纸屑...）
- **背景干扰**：地面纹理、阴影、水渍易误判
- **尺度差异**：从小烟头到大堆垃圾

**推荐技术路径（分级实施）：**

```
Phase 1 (POC):
  - 监督学习 YOLOv8 + 场景化数据标注（5000-10000 张）
  - 先做"显著垃圾堆"，召回优先

Phase 2 (生产):
  - 多模型融合：
    * 目标检测（垃圾袋、易拉罐、纸屑等典型物）
    * 异常区域分割（SAM / U-Net 分割地面异常斑块）
    * 时间维度判断（同位置持续出现 > N 分钟才告警）

Phase 3 (优化):
  - VLM 兜底审核：
    * 不确定样本调用 Qwen-VL / InternVL 二次判断
    * 成本换精度，控制最终误报率
```

> 关键工程难点：垃圾桶旁边正常堆放的垃圾袋 vs 散落垃圾 —— 需要**区域语义建模**（圈出垃圾桶区域作为白名单）。

### 4. 控制误报率 ≤ 5% 的关键设计

误报率是这类项目的**生死线**，纯算法做不到，必须工程组合：

| 手段 | 作用 |
|------|------|
| ROI 区域绘制 | 每路摄像头手动标定关注区域，过滤无关画面 |
| 置信度阈值分级 | 高置信度直接告警，中置信度二次确认，低置信度丢弃 |
| 时空抑制 | 同目标同位置 N 秒内只告警一次 |
| 持续时间判定 | 遗留物 > 30s 才告警，垃圾 > 5min 才告警 |
| 白名单机制 | 工作人员、保洁车辆、固定物体加白 |
| VLM 二次审核 | 边缘检测出告警 → 云端大模型复核 → 推送 |
| 人工反馈闭环 | 误报标注 → 周期性微调模型 |

### 5. 部署形态对比

| 部署方式 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| 纯边缘（盒子部署）| 数据不出园区、合规要求高 | 低延迟、合规 | 算力上限受限 |
| 边缘 + 云（云端管理）| 多站点、需统一管理 | 扩展性强 | 依赖网络 |
| 云原生（视频上云分析）| 摄像头少、轻量场景 | 运维省心 | 带宽贵、有合规风险 |

**推荐**：**边缘推理 + 云端管理**。推理在本地保证实时性和合规，告警/模型/配置走云端统一管理。

## 三、关键风险与对策

| 风险 | 严重度 | 对策 |
|------|--------|------|
| 垃圾检测泛化性差（A 场景训练好，B 场景翻车）| 高 | 现场数据采集 + 微调机制；每个站点 POC 期 2 周采集 |
| 误报率超标导致用户弃用 | 高 | 工程组合拳（ROI + 时空抑制 + VLM 复核）；签合同前 POC 验证 |
| 算力规划不足（实际路数比预期多）| 中 | 售前阶段精确估算：每路约消耗 X TOPS；预留 30% 余量 |
| 现有视频流不稳定（老摄像头 RTSP 断流）| 中 | 自动重连 + 断流告警 + 巡检脚本 |
| 客户数据敏感（住宅/医院不能上云）| 高 | 默认私有化部署，所有 AI 能力本地化 |
| 行为识别精度低于预期 | 中 | 分阶段交付，行为类放到 v2，不写入 v1 SLA |

## 四、推荐落地路径

**MVP（4-6 周）**：入侵检测 + 遗留物检测 + 1 个 POC 站点的暴露垃圾检测
- 目标：跑通端到端链路，验证误报率 ≤ 8%（先放宽）

**生产版（3 个月）**：扩展到全站点，加入垃圾检测的 VLM 复核兜底
- 目标：误报率 ≤ 5%，签订 SLA

**演进版（6 个月+）**：异常行为识别（摔倒、聚集等），按客户需求定制

### 三句话核心卖点

1. 零改造接入存量摄像头 → 节省 70% 改造预算
2. 误报率合同化承诺 → 解决"AI 不好用"的信任问题
3. 边缘部署数据不出场 → 满足合规
